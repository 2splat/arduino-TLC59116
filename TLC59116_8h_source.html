36 
37  This high-level interface tracks each device's state so that commands appear to be independant.
38  For example, you can set the digital-on of a channel without worrying about the
39  other 4 mode settings in each LEDOUTx_Register.
40 
41  Some optimization of the data sent to the devices is also done.
42 
43  ### The TLC59116 Device
44  See the TI spec sheet for much detail: http://www.ti.com/lit/ds/symlink/tlc59116.pdf
45 
46  - Output pins ("channels") are numbered 0..15.
47  - Channels are "sinks", acting like Arduino pins that are LOW, even for PWM.
48  - Connect the minus side of LEDs to the output pins.
49  - Channels are considered ON, or OFF, instead of HIGH/LOW (or, at some PWM value)
50  - PWM is 8 bit.
51  - I2C addresses start at 0x60. See TLC59116_Unmanaged for the available addresses (up to 14).
52  - The device provides a constant-current control (external and software settable).
53  This means you can set the current, usually leave out resistors on each LED/channel, and is more efficient.
54  See also, TLC59116::milliamps().
55  - This library discovers the devices on the I2C bus and keeps them in a list.
56  - Use the _index_ in this library: e.g. "0" is the device with the lowest address,
57  "1" is the next highest, etc.
58  - This library exposes I2C addresses as 7bit addresses.
59  - Individual devices cannot be "reset", all devices on the I2C bus are affected by a reset.
60  - By default, all devices respond to broadcast ("allcall") commands (a special address).
61  - This can be changed, or turned off, for each device (see TLC59116::allcall_address())
62  - Three additional "broadcast" addresses can be set. See TLC59116::SUBADR_address()
63  - Thus, a device can participate in 4 different broadcast groups (or none):
64  - allcall_address()
65  - SUBADR_address(0)
66  - SUBADR_address(1)
67  - SUBADR_address(2)
68  - See the various groupings of the functions (below) for functionality.
69 
70  ### Protocol:
71 
72  - Setting up
73  - Declare a static TLC59116Manager (static scope, or "static" keyword if you are very clever)
74  \snippet allfeatures/allfeatures.ino Need a manager
75  - Initialize the serial-output e.g.
76  - `Serial.begin(9600)`
77  - This library will warn you about some mistakes or dubious usages.
78  - You don't have to setup the serial-output, but then you won't see the warnings.
79  - I like to set the speed to 115200, because at that setting it has little effect
80  on the speed of your program.
81  - Execute \ref TLC59116Manager::init() ".init()" for the manager once, usually in setup()
82  \snippet allfeatures/allfeatures.ino Init the manager
83  - The devices will then be in a ready state (by default, outputs are enabled, but all channels are off)
84  - Obtain a reference from yourmanager\[i], for i from 0..device_count()-1
85  - This library passes and returns references so that "." can always be used.
86  \snippet allfeatures/allfeatures.ino Static referencing devices
87  \snippet allfeatures/allfeatures.ino Index referencing devices
88  - Get information about the device as needed from the \ref Info Functions
89  \snippet allfeatures/allfeatures.ino device info
90  - Set device \ref Global Functions, \ref Group/Broadcast Addresses, \ref Software Current Control, typically in setup()
91  \code{.cpp}
92  void setup() {
93  \endcode
94  \snippet allfeatures/allfeatures.ino device setup
95  - Do initial \ref Digital Functions, \ref PWM Functions, typically in setup()
96  \snippet allfeatures/allfeatures.ino initial channel settings
97  - During loop, mess with \ref Global Functions, \ref Group/Broadcast Addresses, \ref Software Current Control, \ref Digital Functions, \ref PWM Functions
98  - This library returns the device-reference for most calls, so you can chain functions, e.g.:
99  \snippet allfeatures/allfeatures.ino chain functions together

235  /** \name Group Functions
236  \warning The TLC59116 has a hardware bug:
237 
238  If you decrease the value (blink_time or brightness) in these functions,
239  the chip may act like the value is max for one "cycle".
240  This causes a full-brightness result for group_pwm, and a 10 second blink-length for group_blink.
241  I think this is because:
242  - There is a continously running timer, counting up.
243  - The "value" is compared to the timer, and triggers the action and a reset of the timer.
244  - If you move the "value" down, you may skip over the current timer value,
245  so it runs to the maximum value before wrapping around again.
246  "reset" is the only thing that I know of that will reset the timer.
247  But, more experimentation is needed.
